PIPELINE · docs2llm

De documentación cruda a datasets para LLMs

Dos comandos, dos etapas: build-corpus transforma documentación desordenada en un corpus normalizado por chunks; build-qa lee ese corpus y genera pares pregunta-respuesta verificados. Cada etapa es una cadena estricta de módulos — el orden importa y cada uno solo hace una cosa.

Glosario

Los términos que aparecen una y otra vez de acá en adelante.
versión canónica

La carpeta de documentación elegida como "la buena" cuando un proyecto tiene varias versiones exportadas. Se elige a mano, con evidencia de hashes — no por convención de nombre (latest no siempre es la más nueva).

AST

Árbol de sintaxis: representación estructurada de un documento (encabezados, tablas, código) en vez de texto plano. Permite saber dónde empieza y termina cada parte sin adivinar por indentación.

bloque atómico

Una unidad de contenido — una tabla completa, un bloque de código, un párrafo — que nunca se parte a la mitad al armar chunks.

chunk / chunking

Fragmento final del corpus: agrupar uno o más bloques de una misma sección hasta un tamaño razonable. Es la unidad que luego se usa para retrieval o para generar preguntas.

token / tokenización

La unidad mínima que "ve" un modelo de lenguaje — no es una palabra ni un carácter. Se mide con un tokenizador real (tiktoken, encoding cl100k_base), no con una aproximación.

deduplicación exacta

Detectar contenido idéntico (tras normalizar espacios/mayúsculas) calculando un hash SHA-256. Dos fragmentos con el mismo hash son el mismo contenido, sin ambigüedad ni similitud "aproximada".

duplicado aproximado (near-duplicate)

Dos fragmentos que dicen lo mismo pero con texto distinto (ej. una tabla con una columna renombrada) — hash distinto, mismo significado. Se detecta por similitud coseno entre embeddings semánticos, no por comparación de texto.

componente conexa

Grupo de documentos que quedan "atados" porque comparten al menos un fragmento duplicado, exacto o aproximado. Existe para que el split nunca separe ese contenido compartido entre train y test.

split (train/val/test)

Partición del corpus en 3 subconjuntos con roles distintos: train para ajustar, val para validar durante el desarrollo, test para evaluar al final sin haber "visto" antes esos datos.

grounding / evidence_span

Que una respuesta esté sustentada por una cita textual exacta del documento fuente, verificable como subcadena literal. Es lo que separa un dataset confiable de uno que "sonaba bien".

is_low_signal

Marca sobre un chunk demasiado corto o puramente placeholder ("TO DO."). No se borra —se preserva para auditoría— pero se excluye de la generación de preguntas.

QC (control de calidad)

Chequeos automáticos que un par pregunta-respuesta debe pasar antes de entrar al dataset final: evidencia verificada, longitud razonable, no ser casi-duplicado de otra pregunta del mismo proyecto.

tope anti-volumen

Límite máximo de pares Q&A por documento fuente, para que un archivo grande no domine el dataset final solo por tener más secciones que otros.

discovery.py

Descubrimiento de archivos

Recorre solo la carpeta canónica de cada proyecto (la versión elegida en config.yaml, no todas las disponibles). Descarta ruido de navegación de docsify (_sidebar.md, _coverpage.md, _navbar.md) y documentationTypes.json (del cual sí extrae una URL de referencia antes de descartarlo). Clasifica cada archivo restante como Markdown u OpenAPI según su extensión.

Por qué así

Sin este filtro, un archivo de configuración de la UI de docsify terminaría convertido en un chunk de "contenido" — ruido que ni sirve para retrieval ni tiene una respuesta real detrás. Filtrar por nombre de archivo conocido es más simple y predecible que intentar detectar "esto es navegación" por heurística de contenido.

normalize.py + parse_markdown.py / parse_openapi.py

Normalización y parseo

Antes de parsear, 3 fixes puntuales sobre Markdown: encabezados #Titulo sin espacio, entidades HTML doble-escapadas dentro de bloques de código, y líneas de estilo Mermaid repetidas dentro del mismo diagrama. Luego parse_markdown.py arma un AST real vía markdown-it-py (no corta por caracteres) y lo convierte en un árbol de Secciones con bloques atómicos. parse_openapi.py hace lo mismo para swagger.yaml: una sección por endpoint y una por schema, resolviendo $ref a mano.

Por qué así

Se parsea Markdown y OpenAPI con dos parsers separados (no un chunker de texto genérico) porque un swagger.yaml tiene estructura rica — parámetros, enums cerrados, códigos de respuesta — que se pierde si se trocea por líneas: partir un endpoint entre "parameters:" y "responses:" da un chunk que no sirve ni para retrieval ni para generar una pregunta completa. Un AST real (en vez de partir por líneas en blanco) es lo único que garantiza no cortar una tabla o un bloque de código a la mitad más adelante.

chunking.py

Empaquetado en chunks

Bin-packing heading-aware: acumula bloques de la misma sección hasta ~300 tokens (blando), corta a los 512 (duro), y nunca parte una tabla o bloque de código a la mitad salvo que uno solo ya exceda el máximo (ahí cae a una ventana con solape de 60 tokens). Secciones muy chicas se funden con la siguiente; los chunks de OpenAPI nunca se funden. Un chunk resultante muy corto o placeholder se marca is_low_signal, no se descarta.

Por qué así

300 / 512 / 60 no son arbitrarios: 300 tokens es un fragmento con contexto suficiente para responder una pregunta sin arrastrar contenido de otra sección; 512 es el tope duro que evita chunks desproporcionados cuando una sección es rica; 60 (~15% de 512) es solapamiento solo para el caso raro de ventana, para no perder contexto justo en el corte. Nunca partir tablas/código no es una preferencia estética: una tabla partida a la mitad entre dos chunks arruina cualquier pregunta que dependa de ver fila+encabezado juntos.

dedup.py

Deduplicación

Opera sobre el corpus completo, no archivo por archivo, en dos capas. Capa 1 (hash exacto): hashea cada bloque/chunk (espacios colapsados, sin mayúsculas) y marca todo duplicado exacto excepto el primero visto (nunca borra filas). Capa 2 (near-duplicates, embeddings): para contenido que cambió de texto pero no de significado (ej. una tabla con una columna renombrada entre versiones), compara embeddings semánticos (sentence-transformers, modelo all-MiniLM-L6-v2 vendorizado, sin red) con similitud coseno. Las dos capas alimentan las mismas componentes conexas vía union-find: si un fragmento aparece repetido (exacto o aproximado) en varios archivos, esos archivos quedan atados — es la unidad real que usa el split.

Por qué así

El hash es ciego a contenido que dice "casi" lo mismo con texto distinto — de ahí la segunda capa. El hash se calcula a nivel de bloque atómico, no del chunk ya empaquetado — un fragmento repetido que queda mezclado con contexto distinto en cada archivo no se detecta si se compara recién el chunk final, por eso hay una primera pasada de parseo completo antes de chunkear nada. El umbral de similitud (0.9) se calibró de forma empírica contra el corpus real, no copiado de un valor de literatura: los chunks api_spec quedan excluidos de esta capa porque su formato fijo (ver parse_openapi.py) hace que operaciones genuinamente distintas den similitud alta solo por compartir plantilla, no contenido repetido.

splitting.py

Split train / val / test

Asigna 60/20/20 por cantidad de CHUNKS, global sobre todo el corpus (no por cantidad de documentos, no recalculado por proyecto), operando sobre las componentes conexas de dedup.py (nunca las parte). Algoritmo LPT (longest-processing-time-first, un heurístico clásico de balanceo de carga): ordena los grupos de mayor a menor peso (en chunks) y cada uno se lo lleva el split que en ESE momento esté más lejos de su cuota — con el estado compartido entre TODOS los proyectos, no reiniciado en cada uno.

Por qué así

Los documentos de este corpus van de 1 a 51 chunks — repartir "por cabeza" ignora ese peso. La primera versión de este balanceo sí era por proyecto, pero eso resultó un problema real: cada proyecto, evaluado aislado, mandaba "razonablemente" su documento más pesado a train; con 10 proyectos haciendo eso a la vez, sin coordinación entre ellos, train terminaba en ~74% del corpus real. Haciéndolo global (un solo estado de cuota compartido) el resultado real da 59.9/20.1/20.1% — prácticamente exacto.

qa_templates.py + qa_llm_generate.py

Generación de candidatos — dos métodos, a propósito

Hay dos formas de generar un par pregunta-respuesta porque hay dos tipos de fuente muy distintos. Contenido estructurado (una tabla, un endpoint OpenAPI) ya tiene la respuesta ahí, en un campo — extraerla es determinista. Contenido en prosa (reglas de negocio, explicaciones) requiere "entender" texto libre, algo que ningún regex puede hacer de forma confiable — para eso se usa un LLM.

Templates (qa_templates.py)

Regex sobre un formato de texto que el propio pipeline escribió (parse_openapi.py define exactamente cómo se ve un chunk "Endpoint:" o "Schema:"; una tabla Markdown se parsea por columna). Como ambos lados del contrato los controlamos nosotros, el grounding es perfecto por construcción: la respuesta ES literalmente la celda de la tabla.

Ejemplo real — tabla rule_custom_field:

fuente → Q&A generadofila: | id | bigint | Not null | Auto-increment primary key | P: ¿Qué tipo de dato tiene `id` en la tabla de `rule_custom_field Table`? R: bigint (Auto-increment primary key) evidence_span: la fila completa, literal
LLM (qa_llm_generate.py)

Llamada a la API de Claude con tool-use forzado (submit_qa_pairs): el system prompt exige una cita literal (evidence_span) por cada par, y pide devolver lista vacía en vez de forzar una pregunta de baja calidad. No se confía ciegamente en la respuesta: qa_grounding.py la vuelve a verificar después.

Ejemplo real — regla de negocio en prosa:

fuente → Q&A generadofuente: "DELETED: Brand has been permanently removed (irreversible action)." P: ¿Es reversible el estado DELETED de una marca? R: No, es una acción irreversible. (requiere inferir la respuesta, no solo copiarla — por eso hace falta LLM)
Trade-off de correr sin API key build-qa --no-llm (o sin ANTHROPIC_API_KEY configurada) sigue funcionando: se generan igual los pares de tablas/OpenAPI. Pero en esta corrida real, de los 841 candidatos generados, 593 (70%) salieron del LLM contra 248 (30%) de templates — y de las 246 filas finales, 193 (78%) son vía LLM. Sin la API, se pierde casi toda la cobertura de reglas de negocio y explicaciones en prosa, quedando un dataset funcional pero mucho más chico y centrado solo en lo estructurado (tablas y specs).

qa_grounding.py

Control de calidad

Filtro obligatorio: la evidencia debe ser subcadena literal exacta del chunk fuente (no un match difuso) — el gate anti-alucinación central del dataset. Además: longitud razonable de pregunta/respuesta (8-300 caracteres), y descarte de preguntas casi-duplicadas dentro del mismo proyecto (texto normalizado, primera aparición se conserva).

Por qué así

Subcadena exacta y no similitud aproximada: un match difuso no prueba nada real, solo mide que el LLM "suena convincente". Comparar casi-duplicados por proyecto (no global) porque preguntas idénticas en dominios distintos (ej. "¿qué sitios soporta?") son coincidencia de fraseo, no redundancia real.

HallazgoDe 841 candidatos generados, 828 (98.5%) pasaron QC — solo 13 se descartaron. El grueso del filtrado real no ocurre acá: el system prompt del LLM (exigir evidence_span desde el origen) ya evita la mayoría de los casos problemáticos antes de que lleguen a este chequeo.
candidatos → QualityChecks (qc_passed)

qa_build.py — tope anti-volumen

Límite por documento

Máximo 8 pares por documento fuente, repartidos round-robin entre sus chunks (para que el tope no se lo coma el primer chunk) y priorizando, dentro de cada chunk, las respuestas más verificables primero (enum > numérico/booleano > extractiva > generativa).

Por qué así

Sin este tope, un documento con muchas secciones (un swagger.yaml de 35 endpoints, o un README largo) dominaría el dataset final por volumen bruto, no por densidad real de señal — "más volumen no es mejor" es un criterio explícito del enunciado del desafío.

Hallazgo27 de los 34 documentos con al menos una fila de Q&A terminaron exactamente en el tope de 8 — evidencia de que, sin este límite, esos documentos habrían aportado bastante más volumen del que efectivamente dejaron entrar.
candidatos aprobados → qa.jsonl final

Lo que queda garantizado

Sin fuga entre splitsUn fragmento duplicado entre archivos nunca queda repartido entre train y test — se agrupa por componente conexa antes de splitear.
Grounding verificableToda respuesta de Q&A trae una cita textual que se valida como subcadena exacta del chunk — no se confía en que el LLM "dice la verdad".

Librerías utilizadas

Qué es cada una y para qué se usa acá, en palabras simples.
markdown-it-py + mdit-py-plugins

Lee un archivo Markdown y lo convierte en un árbol navegable (dónde empieza cada título, tabla, bloque de código) en vez de texto plano. El plugin agrega soporte para tablas con formato GitHub. Se eligió sobre alternativas como mistune por tener ese soporte de tablas ya maduro.

beautifulsoup4

Limpia HTML embebido dentro de los archivos Markdown (por ejemplo un <div> o <details>) y lo deja como texto plano legible, sin las etiquetas.

PyYAML

Lee archivos .yaml: tanto config/config.yaml (la configuración del pipeline) como los swagger.yaml (las especificaciones OpenAPI de 3 proyectos).

tiktoken

El tokenizador real (encoding cl100k_base) que mide cuántos "tokens" tiene un texto — la misma unidad que cuenta un modelo de lenguaje, no una aproximación de caracteres. El archivo que necesita para funcionar está guardado dentro del proyecto, así nunca hace falta descargarlo de internet.

sentence-transformers

Convierte texto en un vector numérico (embedding) que captura su significado — dos textos parecidos en significado quedan cerca en ese espacio, aunque usen palabras distintas. Se usa el modelo all-MiniLM-L6-v2 (chico, ~90MB, de propósito general) para la segunda capa de deduplicación. También guardado dentro del proyecto, sin depender de descargarlo.

scikit-learn

Se usa solo su función cosine_similarity, para medir qué tan parecidos son dos embeddings — el número que se compara contra el umbral de 0.9 para decidir si dos chunks son casi-duplicados.

anthropic (SDK de Claude)

La librería oficial para llamar a la API de Claude. Se usa para generar preguntas y respuestas a partir de texto en prosa (lo único que un simple patrón de búsqueda no puede "entender" por sí solo).

truststore

Hace que Python confíe en los certificados de seguridad que ya reconoce el sistema operativo, en vez de solo en una lista propia de Python. Hizo falta porque, en redes de oficina con un proxy que inspecciona el tráfico, las llamadas a la API de Claude fallaban por un certificado que Windows sí conocía pero Python no.

python-dotenv

Lee el archivo .env local (donde vive la clave de la API de Claude) para no tener que configurar esa clave a mano cada vez que se abre una terminal nueva.

pytest

El framework que corre los tests automáticos del proyecto — confirma que cada pieza del pipeline (chunking, deduplicación, split, control de calidad) sigue funcionando como se espera después de cualquier cambio.
